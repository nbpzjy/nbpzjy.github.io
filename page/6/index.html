轮胎那些事

从买车开始到现在大约是两年两个月了吧，第一次遇到轮胎被干掉的尴尬

轮胎被干掉

这个问题似乎从来没有找上我，自从买车以来，除了偶尔有些小刮之外，似乎没有遇到什么大问题，于是乎就飘飘然起来了——so easy.

然而，轮胎这个只是概率问题的毛病，总会来的。给我提了一个大大的醒——当心你的轮胎。

在这样一个雨夜，遇到轮胎漏气真真是太不爽了。由于下雨时间好像有些长了，路面积水虽然不严重，但也是有的。七点多，送LP去车站——这就是双城生活的痛点。最近发现走江边似乎真的会快一点，于是就走上了这条路。前面也是有车的，我一般开车都比较稳当，很少抢道加塞之类的，跟着前面的车在走，这条路以前也走过，没有什么大坑，雨下着——注意力都在关注有没有穿插的行人或者电瓶车，速度不快不慢地跟着前车行驶。哐当——似乎压了个坑之类的，由于以前也有过类似感觉，没怎么在意。

大约过了几十秒，一个奇怪的从来没有听到过的声音在提示“右前轮快速漏气”，我勒个叉，中招了，胎压监控报警。从来没遇到过的轮胎被干掉的问题出来了。

说实话，起先是有点慌的：这特么黑灯瞎火，下着雨，并且以前从来没遇到过这样的情况，可怎么搞。靠边停下出去看看，还在嗤嗤漏气呢，不行了。气压也很快由2.5下降到1.3。估计坚持不了了。

好在印象中附近有修理厂，慢速开过去吧。还好没关门，正好有人在。

处理过程

卸下来一看（其实不卸也看到了），任然在快速漏气中。由于装了胎压监控，必须提醒一下，别把这玩意给搞坏了。

放完气，翻出轮胎，说是有轮胎线被搞断了，可以补但是会鼓包了——悲催，这可不好玩。鼓包还能用么？补胎说是得50，而换一个新的500多，重点是前轮不能只换一个，不然左右不一样咋弄。估计是要尽快换两个轮胎了。然而现在临时怎么搞？一个旧轮胎，特么说是要200，据说是以前扎过钉子——补起来的。没办法，只能将就换上旧轮胎——说了半天讲价150.

在这时间段，LP打车也没打着，只能尽快弄好继续前往车站。时间是来不及了，不得已改签，推迟一个小时。

旧轮胎换上轮毂，要做一个平衡——配重什么的搞一下。LP发了好奇心——这是做啥呢？呵呵。

胎压监测怎么不更新

换上了，但是胎压监测怎么还是提示0.9的压力呢？鼓捣一会儿也是这样——起先以为是不是出什么错误或者是，好像要用放气的过程来匹配的——可是没找到。后来提醒说，要跑起来——

好像确实是，于是付钱，走人。跑了一会儿，再看胎压监测，2.6了——谢天谢地，没什么问题。总的来说过些时间还是得换两个前轮了。

学到了什么？

轮胎是个重要的问题，以前好像没怎么重视过，假如在荒郊野外，这怎么搞。换备胎，似乎也不知道怎么换哈。总要学会的。

我觉得应该有几个方案：

  • 勉强开着去修理（比如今天，条件是：近，不严重）
  • 换备胎（上面条件不满足，只能这样了，当然还可以有下面的选择
  • 拖车：
    1. 保险公司拖车，这个应该有空研究一下，保险公司是有几次或者多少公里内的免费拖车的
    2. 拖车公司——这个就坑爹了，估计费用不少，而且还得去修理厂花钱

不管怎么说，总要解决。好在此次事故以花了150块钱买个旧轮胎，推迟LP发车时间为代价——直接导致到杭州时间至少十一点。

听说我们成了纳税百强

听说，我们单位成了纳税百强

牛逼的我们必须牛逼

一不小心，原来我们成了纳税百强企业。难过的是，据说整个集团，我们纳税几乎占了百分之三十——是不是该觉得很牛逼呢？

先看图吧

再来一张

看到这里，我确实觉得我应该为我自己感到骄傲了

  • 首先，我不是什么高风亮节之类
    但是我觉得起码我能为国家做一些贡献了，好歹也跟着交了好多的税嘛。总归是个有用的人，不能归于无用之列了罢。
  • 然后，也说明应该给我发了很多不是，通过税额来反推嘛——虽然并没什么用。但是我也觉得开心。
  • 总感觉现有一个现象：
    规则总是给守规则的人定的。
  • 但是，那又怎么样呢？我喜欢做一个守规则的人，起码我是愿意守规则的。

就好比开车

这得从上次开车去杭州说起了。
由于我的车拿去刷漆去了，正好有有一些原因要去杭州，并且比较急，所以不得已开了_扁哥_的车去杭州了。

大约是周三去的，记得应该是一个夜班回来，大约中午前11点半到的家，放下包直接打车去开车。然后直奔机场，再去的杭州，从开车出发到达杭州总共也就两个小时，15点到了杭州某医院，我们家那位简直惊呆了。

在杭州呆了大约三天——我犯了一个不能饶恕的过错，特么的我的驾照放在我自己的车里了，没拿出来，却开了_扁哥_的车去了杭州——弱弱地问一下，开车需要把驾照带在身上么?
——应该是的吧。

以上都是废话。

我就是担心，在杭州那几天万一有个违章什么的就尴尬了，毕竟开别人的车去呢。所以过了一个礼拜我还是问了_扁哥_那几天有没有违章什么的。他说没收到，应该是没有吧。我说我是遵纪守法的呢。他就呵呵了。

诚然，我觉得借车是个很麻烦的是情——毕竟这个大东西，所牵扯的事情比较多，加入有一些违章啊事故啊什么的，给别人造成的影响会很大。自己的尚且小心，别人的该小心再小心再小心了。

该总结了

其实写这些没什么特别的意思，我就想说我就是一个守规则的人，所以，请祖国放心。

杭州看牙的一点经历

由于门牙有点松动，老婆坚持要在她们的医院联系一下给我看看，所以周五下班就开车到了杭州，准备好好看看牙

先放一个我们家的小朋友照片

情况描述

应该是有大约半年多或者更久一些，就发现两个门牙之间的空隙有点增大，至少在半年前吃玉米苹果什么的，用门牙咬都是没问题的，但是这半年以内竟然门牙有松动了——我的天，这可怎么搞？有过一段时间很痛，那时候只是想只要不痛，松动就松动吧，后来慢慢不痛了，但是晃动是依然的。

虽然也不好看了，但是只要不痛，我忍了。反正一直是个心病留在心里，一直也没有去把这个问题好好处理。

为什么突然要去看牙了呢

起因是最近有个亲戚从老家来杭州看病，孩子的脊柱有点弯曲——通俗描述就是脊柱不直了，由于某种原因，长的不是直的而是弯曲了。我觉得是个很麻烦的事情，毕竟脊柱，弯掉了要想把它弄正，似乎不太可能——当然这是我的理解而已。

但是看了医生之后说是带一个什么支架，通过外面慢慢地矫正。那么我就在想，竟然脊柱都能矫正，那么我一两个牙齿应该还算容易吧？所以就让我燃起了希望，希望能把牙齿问题解决。这才开始谈起了这个问题，老婆也才积极地想怎么处理，有什么熟悉的牙科医生，联系了一圈，所以打算去医院看看，找个医生把情况捋一下，看什么原因导致的，有什么好的办法。

宁波到杭州

周五上午是7点去上班的，回到家大约是11点半，到家的是母亲和豆豆正在吃饭。匆匆地吃完午饭，带豆豆玩了一会儿，等母亲把饭吃完班厨房的事情处理完成之后，应该有十二点多一点左右，就出发了。

此时是下着雨的，很冷。

车没油了，所以在江南路中兴路交叉那里➕满了油——用的是老婆单位办的优惠油卡。然后沿着通途路一直到大隐上的高速。一路无话，到滨江医院时大约是15点了。

看病之路

挂号

按常规，在医院看病当然是先挂号了。由于我的医保卡没有开通异地支付，所以医院的智能终端无法读取我的医保信息。回去得好好咨询一下，异地结算权限是怎么回事，怎么开通——或者说能不能开通。

所以医保挂号是行不通了。

只能办一个医院的就诊卡了。办卡倒是容易的，直接在智能处理终端，可以申请办卡，只需要身份证，就可以办理了，一元钱的成本，事实上我以前应该是办过卡，所以我的身份证读取之后，我的账户里面还有9块钱。

港区航行的“早、大、宽、清”

21日上午有幸听到18#康老师的一堂关于港区航行的速度控制的课，收获颇丰。对于港区这个特殊水域中，怎么么做才是安全，有了一些新的认识。仔细地梳理了在国际海上避碰规则中有关速度相关的一些规定及描述，这是一种新的阅读和理解规则的方法。于是我就有了一种，怎么去理解规则中的早大宽清，在港区航行中怎么才能符合这个规定

大幅度及早采取行动宽让他船直到完全让清，这就是早大宽清的由来。

在港区，尤其是宁波港这样的航道极为繁忙，交通密度大，又处于岛礁区域，潮流复杂，且有很多遮蔽区域，对于怎么做到早大宽清，有了更为复杂的要求。

规则，是关于两个船舶直接的相互关系来做描述的，事实上有三条或者以上的相关船舶的局面已经属于特殊情况了。而通常为简化操作，往往把复杂局面分解为相互有影响的两两相遇的局面。

  1. 虽然很难做到早、大、宽，但目的必须是清。
  2. 怎么样做到让的适当，这是一个很大的课题。不同类型的船舶，不同尺寸的船舶，以及操作人员不同的操作水平，都会导致怎么让的合适有着较多的区别。

有过一些嚷得过多而造成与其他船舶发生碰撞的教训

某日一条出口船在羊小猫北出口航行，由于能见度不良，进口小船的航行动态不明朗，导致在避让的过程中与周边的挖砂船发生了 擦碰擦，造成了一定的经济损失，特别是给当事人造成了深远的影响。

DaVinci出现RED Decoder错误

先上图

之前安装好的DaVinci12.5，版本是Resolve，iMac运行正常，并且学习了一级调色（何清超的初级教程）。然而今天竟然出问题了，在登录之后出现一个错误：
Failed to initialize RED Decoder with the configured number of decoders.
虽然在点击OK之后会进入工程，似乎各种操作也没什么问题，但是这样每次打开都出错也让人很无语啊

首先当然是祭出Google搜索大法

我也学会了在能不用度娘的情况下尽可能不用，最好google。直接原文粘贴过去搜。得到的搜索结果也蛮多。

解决方法

  1. 有说要买正版的——关键是Resolve版本本来就是正版啊，只不过属于免费版本，又没有做什么破解一类的。这个办法让我无语。

    MAC系统达芬奇提示错误的解决办法
    Failed to initialize RED Decoder with the configured number of decoders.
    Error: Decoder client version (11.1.0.023) does not match RED Decoder version (11.1.2.013)
    mac系统出现以上错误 是因为没能正常启动 编码解码 导致的，是编码器问题， 引起这种问题的 原因是使用了破解版的 达芬奇软件导致编码无法启动，请购买正版 插入加密狗可解决

  2. 有说下载工具的，可是我并没有什么下载工具啊。——后来才想起来我的百度云在下载东西，此事后话。

  3. Q:遇到错误提示“ Failed to initialize RED Decoder with the configured number of decoders”怎么办？装了REDCINE-X也不管用。关闭警告后好像不影响使用。
    A:这么看来应该不是REDCINE-X的问题了。这个问题在BMD官方论坛上也讨论过。有的人说重装达芬奇，有人说新建工程再试试。有人说是IP端口冲突造成的。官方说请上传达芬奇的LOG文件供他们分析。因此目前还没有一个定论。英文好的可以参考这个链接：http://forum.blackmagicdesign.com/viewtopic.php?f=3&t=528
    另外，如果你是Windows系统，防火墙可能会干扰DpDecoder通信。可以尝试关闭防火墙。看看是否问题解决了。另外，如果你使用BT下载软件，例如Utorrent之类，也可能干扰通信，关闭之后再试。
    附上下载地址：REDCINE-X_PRO_Build_33【Windows版】
    http://pan.baidu.com/s/1sjmICWl

总体来讲都还没解决

而这里却能找到解决办法。

重点在执行 sudo lsof -iTCP: 10000 -sTCP: LISTEN
这里是查看10000这个端口是什么程序在监听
看，有两个叫BaiduNeted PID 2151的在监听这个端口，而这个就是问题的关键了，百度云的端口跟RED Decoder的端口冲突了，到至了上文提到的那个错我。解决办法是停止百度云——就这么简单。停止百度云之后再查看这个端口是没有程序在监听的。然后打开DaVinci就正常了，没有上文的那个错误了，并且再去查看10000端口，是一个DPDcoder PID10087的程序在监听这个端口了

跟着96去光明

有很多时间没有写关于引航的小文章了，这里打算开始写，尽量多总结，多写东西，多思考，也许有一天我也会变得厉害起来呢，谁知道会不会呢？至少可以尝试把需要完成的一篇文章先搞定。奔着这个目标去也应该奋斗

码头情况

计划是1630靠泊光明，按常规是1430虾峙门东进口。船长225米，吃水约为13米左右，属于重载。

光明码头，在宁波港的众多码头之中，由于其难靠而比较闻名，属于位置极差潮流极复杂前沿船舶交通密度极大的一个区域。

靠泊时间及该时段的相关情况

时间

本次跟船学习，靠泊计划是1630，镇海高潮时间是1715，也就是大约在高潮前三刻钟左右。

这个时间段的特点

由于在镇海高潮前，所以由虾峙门进口至接近码头，均为涨水时间段。这对于重载船舶慢速主机来说，对于提速的需求是容易实现的，至少不用太过担心太慢而迟到的问题，反而应该注意过早到达相应水域，控制速度。也是因为顺流的原因，在船舶减速阶段要提早开始，或者应该充分考虑到顺流对减速带来的困难。

由于航道在涨流，所以在航道航行的小型船舶，在这个时间段应该基本是进口，由于接近涨流的后期，需要进口的小型船舶大多已经进去。在这个时间段即使有船也应该相对偏少，这一点对于需要在减速阶段与其他船舶会遇的情况来说是有宜的。当然也还要看具体情况，并且也不排除有些不按常规进出的船只。总的来讲，还是需要相互沟通协调，特别是在这种需要他船的配合或者甚至完全靠他船来避让的情况，沟通显得尤为重要。

靠泊过程

航行阶段

进口时间大约1422，也基本是按照计划来进行。到达羊小猫东的时间大约1530前后，从虾峙门到这个位置消耗一个小时，属于基本常规做法。并且从这个位置到靠泊计划时间还剩一个小时，也是非常舒服的，不早不晚。

拖轮要提早联系，最好在虾峙门上船时就先咨询一下，所需拖轮的前序作业情况，以及停靠位置。在峡子门出来时需要联系确认大约半小时要到达指定位置。

由于在羊小猫东要开始减速，且左转调整姿态，在潮流复杂交通流复杂的情况下，早一些带好拖轮，是必须的也是符合要求的。

在羊小猫东开始控制速度，至羊小猫北大约8-9节，慢慢左转，航行在出口航道的南面，速度进一步慢慢衰减。

带拖轮

为了控制速度及协助控制位置调整靠泊姿态，羊小猫北就应该带好。一个在又前第二个位置（留着第一个位置给另外一条），一条带右后，再一条带正后方主要用于协助控制速度（前面讲到顺流对于控制速度的困难，在不采用停车——会失去舵效的方法时，有一个拖轮在正后方是一个不错的办法），在完成控制速度之后，即可解掉换至右前第一个位置。

通常放在正后方的哪一个拖轮应该选择马力最大的，所以在留着第一个位置给最大的拖轮，在控速阶段完成，进入靠泊过程时也能发挥最大那个拖轮的作用。特别在靠泊前期控制船头外偏及后期控制横向速度时，其最大马力的作用都极为重要。

靠泊中特别需要注意

由于主航道在涨流时间段，而码头边落流（靠泊也选择左靠），所以就存在一个涨流至落流的变化区域——切变。特别应该注意的是速度的变化过程，在纵向的和横向的都应该特别关注，还有就是偏转的趋势保持，不应该大幅度变化。

总结

总体来讲，我认真地关注船舶的运动状态及师傅的指令了，基本也理解所做的操作的原因，但由于这个特殊码头的特殊性，还需要多学习，特别是在速度的控制，转向过程中的姿态调整，以及很重要的与其他船舶的交汇，沟通协调，拖轮的指挥并发挥其最大作用。

Hexo新建页面踩坑记

开始用Github+hexo部署静态博客有一小段时间了，虽然没怎么写文章，但是部署是成功的。之前是在windows上部署的。最近发现一个叫iPic的图床工具，对于Markdown这样的标记语言的图片展示困难户来说，简直就是救星啊。可惜只找到Mac版的，虽然也看到一个牛人自己写了Mpic作为windows版本的图片上传工具，可惜没有exe文件，自己编译计算了——其实是不会。
所以打算由windows转到mac来写博客的东西。

本来以为直接copy到优盘拿过来就可以了，不过出了很多问题。猜想应该是直接在优盘里不太行吧？本地预览是可以的，没有问题，写文字，generate，都没有问题，所有命令都完美。但是hexo d，就是运行不成功。尝试很多方法，就是推送不出去。npm install， hexo init，都搞了好几遍。没用。最后是windows上面把hexo的源文件整个推到github，再在mac上把它clone下来。问题还是接踵而开。

配置文件及主题文件

不知道怎么回事，配置文件和主题文件都没有过来。猜想大约是gitignore里面设置的问题，这两个东西根本没用推送到github上面去。

无所谓了。反正copy缺少的东西过去。就搞定了。

文章无法生成

这个就坑爹了
特么hexo new “xxxx”成功，然后在_post里面编辑相应文件之后，再回到shell执行hexo g，不行。

红红的error，真特么辣眼睛。

放上空格就搞定了。

hexo d的问题

也是一个红红的error
不过这是小意思啦。不就是npm install hexo-deployer-git –save
就搞定了啊。
